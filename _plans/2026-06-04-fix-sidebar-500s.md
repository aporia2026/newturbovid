# Fix sidebar HTTP 500s: batch the polling + correct the SQLite transaction bug

Date: 2026-06-04
Status: proposed (awaiting approval)
Owner: Yoav
Investigation transcript: end-of-day 2026-06-03 / start-of-day 2026-06-04 session

## Problem (verified, not guessed)

Bulk users see frequent HTTP 500 toasts in the sheet sidebar — on submit, on
kill, on routine polls. The Apps Script already retries 5xx three times; users
still see the error.

### What we ruled out (with evidence)
- **SQLite lock contention.** PA `error.log` over 16 hours: zero
  `OperationalError` tracebacks. If our app threw, we'd see it.
- **PA ASGI beta recycling.** Deployment is WSGI (manual config) wrapping the
  ASGI app via `a2wsgi`, per `pythonanywhere/wsgi_fallback.py`. Not the ASGI
  beta path.
- **Worker dead / not draining.** Always-on task is in `Running` state
  (`/home/Aporianetworks/bulkvid` → `bulkvid.worker`).

### What the evidence actually shows
- `server.log` shows `verify_ok` at a sustained **~1.5–2 req/sec, 24/7**.
- 200 lines of `tail` covers only **90 seconds** — that's how dense the auth
  traffic is.
- All from a small number of users with the sidebar open.
- No tracebacks anywhere in either `error.log` or `server.log` for the 500s.

### Why this causes 500s
PA Developer tier gives a small uWSGI worker pool (≲4 workers). The sidebar's
per-cycle pattern fires **five separate authenticated requests every 3
seconds** while active:

```
listJobs()                          → 1 auth call
getJobRows(job_a), getJobRows(job_b), getJobRows(job_c)  → N auth calls
getJobLog(job_with_open_pane)       → 1 auth call per open log
```

With 2–3 active jobs and a single open log pane, every poll cycle = 4–5 auth
calls. Multiply by polling cadence (3s active) and concurrent sidebar
sessions, and the worker pool occasionally saturates. PA's frontend then
returns **HTTP 500 to requests it can't dispatch to a free worker in time** —
which is exactly the failure mode that produces no Python-side traceback.

## Goal

Eliminate the 500s by making the sidebar's polling pattern cost ~5× less
backend work per cycle, and fix the related-but-different SQLite transaction
bug surfaced during investigation.

## Approach

### Change 1: Batched poll endpoint `GET /jobs/poll`

Single authenticated request that returns everything the sidebar needs:

**Request** (query params):
- `limit` (int, default 100): max jobs to return — same as today's listJobs.
- `logs` (comma-separated job IDs, optional): which jobs' log tails to
  include. Caller sends only the panes currently open in the UI; empty/missing
  ⇒ no logs included.
- `log_tail` (int, default 200, max 2000): lines per included log.

**Response**:
```json
{
  "jobs": [ /* same shape as current GET /jobs */ ],
  "rows_by_job": {
    "<job_id>": [ /* same shape as current GET /jobs/{id}/rows */ ]
  },
  "logs_by_job": {
    "<job_id>": { "exists": true, "lines": [ "..." ] }
  }
}
```

**Ownership rules** (identical to existing endpoints):
- Jobs filter same as `list_jobs` — bulk user sees own, admin sees all.
- `rows_by_job` only populated for jobs in `jobs` (so a malicious `logs=` query
  param can't probe other users' jobs).
- `logs_by_job` filtered the same way.

**Optimization**: only include rows for jobs that are currently `running`
(matches the sidebar's existing behavior — queued jobs don't get per-row
detail; completed/failed/killed jobs are archive rows that don't need refresh).
Cuts the per-poll DB work further when there's a long archive list.

### Change 2: Rewrite Sidebar polling to use one endpoint

In `apps_script/Sidebar.html`:
- Replace `refresh()` flow: instead of three separate `google.script.run`
  calls, fire one `pollAll(Object.keys(_openLogs))`.
- Add `Code.gs#pollAll(openLogIds)` that proxies to `GET /jobs/poll?logs=...`
  via the existing `_fetchJson` (retry + auth header already there).
- Wire response into the existing render path: `_lastJobs`, `_rowsCache`,
  `_logCache`. Render code does not change.
- Slow the **idle** poll cadence: 12s → 30s (when no jobs active). Active
  cadence stays at 3s.

### Change 3: Fix SQLite transaction-boundary bug in `queue.py`

`JobQueue.__init__` opens the connection with `isolation_level=None`
(autocommit). The code then uses `with self._conn:` blocks (`_enqueue_sync`,
`_record_result_sync`, `_kill_job_sync`, `_kill_all_sync`,
`_recover_orphaned_rows_sync`) as if they were atomic multi-statement
transactions. They are not: in autocommit mode, each statement commits
individually, and `with conn:` only issues an implicit `COMMIT`/`ROLLBACK`
that's effectively a no-op since there's no open transaction.

Consequence: the worker can interleave between statements in a "transaction",
leaving the DB in inconsistent intermediate states (e.g. job row inserted but
row_queue rows not yet, or counters updated against a job another process
just killed).

**Fix**: wrap each multi-statement block in explicit `BEGIN IMMEDIATE` /
`COMMIT` / `ROLLBACK` around the existing `with self._conn:` body. Use
`IMMEDIATE` (not `DEFERRED`) so the write lock is acquired up front, avoiding
mid-transaction SQLITE_BUSY.

Smallest-change implementation: a tiny `_tx()` context manager that does
`BEGIN IMMEDIATE → yield → COMMIT/ROLLBACK`. Replace `with self._conn:` with
`with self._tx():` at the existing sites. No behavior change beyond actual
atomicity.

## Alternatives considered and rejected

1. **MySQL on PA / move off PA.** Real options, both flagged in earlier
   investigation. Rejected: the evidence does not point to SQLite contention.
   No `OperationalError` in any log. Migrating a healthy DB to "fix" a
   request-volume problem solves nothing and costs days.
2. **Increase Apps Script retries (3 → 6, longer backoff).** Rejected: this
   hides the symptom and adds latency. Doesn't reduce backend load. The fix
   should remove the cause.
3. **Server-side SQLite retry + `OperationalError` → 503 exception handler.**
   Worth doing anyway as defense-in-depth, but rejected for this change set.
   No evidence of lock errors today; adding a layer for a problem we can't
   reproduce muddies the diagnosis if 500s reappear.
4. **Drop the existing `listJobs` / `getJobRows` / `getJobLog` endpoints in
   this change.** Rejected: we don't have proof nothing else uses them.
   Confirm via PA access log a week after `/jobs/poll` lands, then deprecate
   as a separate change. Cost of keeping them is near-zero (small handlers,
   shared helpers).

## Security & safety (Rule 13)

- New endpoint uses the same `get_identity` dependency as the rest of `/jobs`
  — no new auth surface.
- Ownership check: user only sees jobs they own; admin sees all. Identical to
  `list_jobs` filter logic.
- `rows_by_job` and `logs_by_job` are keyed by job IDs from the *filtered*
  jobs list — a malicious `logs=other_user_job_id` cannot leak rows or logs.
- Log file path sanitization unchanged (uses existing `read_job_log_lines`,
  which already strips `..` and slashes).
- `log_tail` capped at 2000 (matches `read_job_log_lines`).
- `logs` query param length cap (say 50 IDs, ~2KB) to prevent absurd queries.
- No PII or token logged in the new code path beyond what existing endpoints
  already log.
- Transaction fix in `queue.py` strictly improves correctness; no auth or data
  exposure change.

## Testing (Rule 18)

New `tests/test_routes_jobs.py::test_poll_*` cases (project uses pytest +
FastAPI TestClient based on existing route tests):
- `test_poll_owner_basic`: user sees only their own jobs in `jobs`.
- `test_poll_admin_sees_all`: admin sees all users' jobs.
- `test_poll_rows_only_for_running_jobs`: queued/completed jobs don't appear in
  `rows_by_job`.
- `test_poll_logs_filter`: `logs=` includes only requested + owned job IDs.
- `test_poll_logs_unowned_ignored`: requesting another user's job id in
  `logs=` returns no log for it (not 403; just silently absent — matches
  existing list semantics).
- `test_poll_log_tail_cap`: `log_tail=99999` is clamped to 2000.
- `test_poll_empty_state`: user with no jobs returns
  `{jobs: [], rows_by_job: {}, logs_by_job: {}}`.
- `test_poll_logs_param_too_long`: >50 job IDs in `logs=` returns 400.

For the queue transaction fix:
- `test_queue_enqueue_atomicity`: simulate a kill mid-enqueue; confirm we
  never see a job with the wrong row count (currently can intermittently fail
  under load — needs threading or sqlite-level test).
- `test_queue_record_result_atomicity`: similar — counters always match
  `row_queue` row statuses.
- The existing 395-test suite must stay green; rerun all of `tests/` before
  ship.

Manual:
- Run web app + worker locally, submit a multi-row job, open the sidebar (or

  request per 3s instead of 5, response body has all three sections, sidebar
  renders identically.
