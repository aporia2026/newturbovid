# kie.ai body-code 429 → cooldown + retry-other-keys

**Date**: 2026-06-11
**Owner**: Yoav
**Status**: Approved (Option A, no council)

## Problem

In the `simple_x4` batch run on 2026-06-11, rows 27, 28, 30 (and several
more) failed with:

```
All image backends failed. nano-banana-2 + gpt-image-2 (kie) and AtlasCloud.
gpt-image-2=kie.ai submit body code != 200: {'code': 429, 'msg': ...}
```

Root cause: kie.ai signals per-key rate-limit by returning **HTTP 200 with
body `{"code": 429, "msg": ...}`** — NOT HTTP 429. The current
`KieClient.create_task` only catches the HTTP-status 429 path
([kie.py:225-227](../src/bulkvid/adapters/kie.py#L225-L227)). The body-code
path ([kie.py:233-234](../src/bulkvid/adapters/kie.py#L233-L234)) raises a
generic `KieError` and the key is **never placed on cooldown**.

Consequences observed:
1. The just-tripped key stays "available" in the pool. The next call
   (the `gpt_image_2` fallback in
   [image_gen.py:60-66](../src/bulkvid/pipeline/image_gen.py#L60-L66))
   immediately re-acquires the same key and 429s again.
2. With multiple rows running in parallel, every other row also keeps
   hammering the same rate-limited key.
3. The whole fallback chain (nano-banana-2 → gpt-image-2 → AtlasCloud)
   collapses for an entire batch of rows.

## Goal

Treat body-code 429 the same as HTTP-status 429: cool the offending key
down, then try the next available key in the pool before giving up.

## Approach (Option A — chosen)

Two surgical changes in
[src/bulkvid/adapters/kie.py](../src/bulkvid/adapters/kie.py):

1. **Extract single-key submission** into `_submit_once(model, input, key)`.
   Detect body code `429` the same way as HTTP 429: call
   `pool.mark_rate_limited(key)` and raise `KieRateLimitError`. Body's
   `msg` field is preserved in the error message for debugging.

2. **Add retry-other-keys loop in `create_task`**. On `KieRateLimitError`,
   re-acquire from the pool (which round-robins, skipping cooled keys) and
   retry. Hard cap: `min(pool.key_count, 3)` attempts. If all attempts
   trip 429, the last `KieRateLimitError` propagates — `edit_with_fallback`
   then moves on to AtlasCloud as designed.

Expose `KiePool.key_count` so the cap derives from real pool size (a
1-key pool can't retry; a 2-key pool tries each once; 3+ keys cap at 3).

### Alternatives rejected

- **B (cooldown only, no retry)** — simpler, but every row burns a full
  nano-banana-2 attempt before falling through to gpt-image-2. The
  same key is still acquired across parallel rows during the cooldown
  race window, and the row latency cost adds up across a 200-row batch.

- **C (global pool cooldown when all keys 429)** — overkill; a per-key
  cooldown plus the retry loop already handles the "all keys 429" case
  by raising after the cap, and the global state would tangle with
  parallel row processors. Keep the per-key model.

- **Lower the per-pool cooldown to 5-10s** — masks the bug without
  fixing it; still re-uses a known-bad key inside the same row. Not done.

## Out of scope

- Gemini TTS `429 RESOURCE_EXHAUSTED` (row 32). Separate issue: the
  Vertex AI per-minute quota is the bottleneck and the existing TTS
  semaphore + retry already partially handle it. Will track separately
  if it keeps recurring.
- AtlasCloud retry/cooldown. Adapter today raises a generic `AtlasError`
  on any non-200; no key pool yet so cooldown is N/A. Revisit if Atlas
  becomes the bottleneck.
- Blocking behavior of `pool.acquire()` when ALL keys are cooled.
  Today the fallback chain waits up to `cooldown_seconds` (60s) before
  gpt-image-2 can acquire. Tolerable; revisit if it hurts batch latency.

## Security (rule 13)

- No new attack surface. Adapter-internal change only.
- Error message includes the body `msg` field — kie returns operator-safe
  text here ("rate limit exceeded" etc.); not PII or secrets.
- Key suffix (last 12 chars) is logged on cooldown as today — already in
  the existing 429 log. Full keys never logged.

## Observability (rule 14)

New / changed logs (all under existing `[kie]` namespace):

- `kie_submit_body_code_429` — emitted when body-code 429 is detected.
  Fields: `model`, `key_suffix`, `body_msg` (truncated to 200 chars).
- `kie_submit_retry_after_429` — emitted before each retry attempt.
  Fields: `model`, `key_suffix` (the one that just failed),
  `attempt`, `max_attempts`.
- `kie_key_cooldown` — existing log, also fires on body-code 429.

Existing `kie_submit_ok` still fires on the eventual success.

## Settings (rule 15)

No new user-facing settings. The retry cap is hard-coded (`min(pool, 3)`)
because it's an internal failure-recovery mechanism, not a knob a user
would meaningfully tune. The existing
`KIE_RATE_LIMIT_COOLDOWN_SECONDS` env var still controls the cooldown
window.
